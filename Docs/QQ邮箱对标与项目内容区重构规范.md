# QQ 邮箱对标与项目内容区重构规范

> 项目：复旦大学附属儿科医院技能培训中心 PC 端  
> 文档状态：第一版，作为后续界面重构的结构与验收基线  
> 更新日期：2026-06-27  
> 适用对象：产品、设计、前端开发及后续执行型 AI

---

## 0. 结论先行

当前项目的问题不是“某些页面还不够精致”，而是缺少一套所有页面共同遵守的内容区语法。

已有 `shared/design-tokens.css`、`shared/components.css` 和 Arco Design 覆盖样式，但这些规范目前主要停留在“变量和零散组件”层，没有进一步约束：

1. 一级页面、详情页面、创建/编辑页面分别采用什么结构；
2. 标题、Tab、搜索、筛选、对象操作、视图控制分别放在哪里；
3. 页面正文、卡片、表格、辅助栏应该使用什么字号和间距；
4. 一个模块在重构时如何确认业务信息和操作没有丢失；
5. 不同模块脚本如何保证只有当前模块拥有内容区，不产生空白页或互相覆盖。

因此本轮不应继续逐页“微调 CSS”。正确顺序应是：

```text
先冻结业务信息
→ 建立页面层级和内容区语法
→ 建立通用组件
→ 整改高完成度页面
→ 整改结构尚可但视觉不统一的页面
→ 以旧版为业务底稿重构存在信息或功能损失的页面
```

QQ 邮箱最值得借鉴的并不是蓝色外观，而是：

> 全局框架稳定，页面结构可预测；统一的是信息语法，不是强迫所有业务使用同一种内容形态。

## 2. QQ 邮箱的信息架构

### 2.1 四层产品架构

```text
QQ 邮箱
├─ 全局层
│  ├─ 品牌
│  ├─ 全局搜索
│  ├─ 账户
│  └─ 设置
├─ 一级业务层
│  ├─ 邮件
│  ├─ 文件与附件
│  ├─ 联系人
│  ├─ 日程与记录
│  └─ 内容与创作
├─ 应用内导航层
│  ├─ 文件 / 中转站 / 发票
│  ├─ 全部附件 / 收藏附件
│  ├─ 日历 / 日程
│  └─ 简历 / 我的资料
└─ 当前任务层
   ├─ 浏览
   ├─ 筛选
   ├─ 选择与批量处理
   ├─ 阅读详情
   ├─ 创建与编辑
   └─ 设置管理
```

左侧栏只负责“进入哪个业务空间”，不会无限承载所有业务层级。更细的同级视图通过以下方式出现：

- 内容区顶部 Tab；
- 内容区内部次级导航；
- 右侧上下文栏；
- 分组标题；
- 筛选与视图控制。

### 2.2 内容区的标准插槽

QQ 邮箱的页面可以统一抽象为：

```text
ContentShell
├─ PageNav        返回、面包屑、同级 Tab
├─ PageHeader     页面标题、对象标题、数量、状态
├─ ActionBar
│  ├─ Primary     创建、上传、提交等主操作
│  ├─ Object      删除、移动、下载等对象操作
│  └─ View        搜索、筛选、排序、视图切换
├─ ContextBar     风险、翻译、筛选条件、状态说明
├─ MainContent    列表、表格、卡片、正文、日历、表单
├─ Aside          当前对象详情或上下文辅助信息
└─ EmptyState     空状态与下一步动作
```

不是每个页面都必须出现全部插槽，但同一个插槽不能在不同页面承担完全不同的职责。

### 2.3 页面原型

QQ 邮箱使用六种主要页面原型：

| 原型 | 典型页面 | 核心结构 |
|---|---|---|
| 集合列表页 | 邮件、文件、附件、联系人 | 标题/Tab + 工具栏 + 表头 + 数据行 |
| 分组列表页 | 收件箱、发票、日程 | 时间或状态分组 + 组内对象 |
| 主从分栏页 | 邮件阅读、日历、简历 | 主内容 + 当前上下文详情 |
| 沉浸阅读页 | 邮件全文、记事本 | 返回/操作 + 对象信息 + 连续正文 |
| 编辑工作台 | 记事本编辑 | 退出/保存 + 编辑工具 + 编辑画布 |
| 管理设置页 | 通用设置、应用管理 | 分类导航 + 标题 + 分组表单/表格 |

QQ 邮箱没有把所有内容都做成卡片。它根据对象选择列表、表格、正文、时间网格、卡片网格或画布。

### 2.4 工具栏的信息语法

QQ 邮箱的工具栏表面上动作很多，但顺序稳定：

```text
[位置/范围] [对象操作] [更多]                 [数量/搜索/筛选/视图]
```

常见规则：

1. 返回、上一项、下一项属于导航；
2. 删除、移动、下载、回复属于对象操作；
3. 搜索、筛选、排序、视图切换属于“怎么看”；
4. 创建、上传等最重要动作使用主按钮；
5. 普通操作使用中性按钮；
6. 低频动作进入“更多”；
7. 搜索和视图控制通常靠右；
8. 进入详情或编辑状态后，工具栏会随任务切换，不继续保留列表页的全部控制。

### 2.5 为什么它“看起来复杂，实际有序”

1. **外壳不动**：顶栏、侧栏、内容容器和主要对齐线稳定。
2. **任务状态明确**：用户能立即判断当前在浏览、阅读、编辑还是设置。
3. **信息密度集中**：高密度只发生在主内容区，外围保持克制。
4. **主要依靠位置建立层级**：少用大色块、重阴影和多重卡片。
5. **一个页面只有一个视觉主角**：邮件正文、月历、列表或文档画布始终占最大面积。
6. **右侧栏表达当前上下文**：不会因为右边有空位就塞入无关信息。
7. **认知统一高于像素统一**：不同业务页面允许不同布局，但相同语义始终位于相似位置。

## 3. 从 QQ 邮箱 HTML 提取的视觉基线

以下数值用于理解 QQ 邮箱如何建立清晰度，不要求机械照搬。

### 3.1 文字颜色

QQ 邮箱不是大量使用浅灰，而是使用带蓝灰倾向的黑色透明阶梯：

```css
--base_black:    #000000;
--base_gray_100: #13181D;
--base_gray_080: rgba(22, 30, 38, 0.8);
--base_gray_060: rgba(24, 36, 48, 0.6);
--base_gray_040: rgba(25, 41, 59, 0.4);
--base_gray_020: rgba(23, 45, 69, 0.2);
--base_gray_010: rgba(22, 46, 74, 0.1);
--base_gray_005: rgba(20, 46, 77, 0.05);
--base_white_100: #FFFFFF;
--accent_blue:   #0F7AF5;
```

清晰的原因并不是所有字都大，而是：

- 主要文字接近 `#13181D`；
- 次要文字仍有足够对比度；
- 很浅的灰色只用于真正的辅助信息；
- 12px 不与极浅灰同时大面积出现。

### 3.2 字号与行高

QQ 邮箱常见字号阶梯：

| 用途 | 字号 / 行高 |
|---|---|
| 极小控件 | 11 / 14 |
| 小按钮、标签 | 12 / 16 |
| 紧凑控制 | 13 / 18 |
| 主体列表、按钮、正文辅助 | 14 / 20 |
| 强调正文、重要控件 | 16 / 22 |
| 模块标题、弹窗标题 | 18 / 24 |
| 大标题 | 24 / 32 |

真正的内容主体主要建立在 14–16px 上，11–12px 只服务于小控件或低优先级元信息。

### 3.3 按钮高度

QQ 邮箱按钮形成完整比例关系：

| 高度 | 字号 | 横向内边距 | 圆角 |
|---:|---:|---:|---:|
| 20 | 11 | 4 | 2 |
| 24 | 12 | 6 | 4 |
| 28 | 13 | 8 | 4 |
| 32 | 14 | 8 | 4 |
| 36 | 14 | 10 | 6 |
| 40 | 16 | 12 | 6 |

常规 PC 操作主要使用 32px 或 36px。极小按钮没有被当作普通页面按钮滥用。

### 3.4 间距与圆角

高频间距以以下数值为主：

```text
4 / 8 / 12 / 16 / 24 / 32
```

高频圆角：

```text
按钮与输入框：4–6px
普通容器：6–8px
特殊大容器：12px
```

QQ 邮箱不会在同一视觉层连续嵌套多个有边框、有圆角的卡片。

### 3.5 可借鉴但不应照搬的部分

本项目用户包含医院管理者和业务操作人员，且客户已多次反馈文字偏小。因此项目应比 QQ 邮箱更保守：

- 正文和表格内容不得以 12px 为默认值；
- 普通按钮统一使用 14px；
- 页面标题应高于 QQ 邮箱部分紧凑列表页；
- 医疗业务的重要状态和数字必须提升对比度；
- 不以“高密度后台”为目标压缩可读性。

## 4. 当前项目现状诊断

## 4.1 已有基础与实际断层

项目已经定义：

- 颜色 Token；
- 4px 间距阶梯；
- 11–24px 字号阶梯；
- 共享卡片、工具栏、表格和状态组件；
- `.app-page-toolbar` 与 `.app-detail-toolbar`；
- Arco Design 统一覆盖。

但实际页面仍然存在三套并行体系：

1. 共享组件体系；
2. Arco 原生组件体系；
3. 各模块自定义的 Header、Toolbar、Button、Card 和字体规则。

结果是“变量统一了，页面结构仍未统一”。

### 4.2 当前顶部结构不稳定

#### 工作台

- 标题与“全部服务”直接贴在内容区顶部；
- 搜索框另起一行；
- 顶部区域没有形成清晰的 Header/Toolbar 分层；
- 分类导航、常用服务和全部服务均使用同等强度，首屏缺少明确主任务。

#### 课程开发列表

- “课程 / 专项课程”Tab 代替了页面标题；
- 搜索位于第一行右侧；
- 视图切换落到第二行最左；
- 四个筛选、重置和两个创建按钮挤在同一行；
- 对象范围、视图控制、筛选和对象创建混在一起。

#### 上传/生成教案

- 返回按钮单独占据一个位置；
- 页面标题与返回按钮之间缺少统一关系；
- 三种创建方式只有大卡片，没有步骤、选择状态和明确的后续流程；
- 禁用、次操作和不可用状态不够容易区分。

#### 课程详情/编辑工作台

- 返回与对象标题在第一行；
- 右侧栏折叠按钮掉到下一行，形成“飞起来”的孤立控件；
- 详情、编辑入口、步骤导航、资源管理、AI 助手和底部发布操作同时存在；
- 页面既像详情页，又像编辑器，还像配置工作台；
- 右栏承担课程状态、考核、资源等多种职责，信息负担过重。

### 4.3 文字问题

当前全局 Token：

```css
--font-xs: 11px;
--font-sm: 12px;
--font-md: 13px;
--font-base: 14px;
--font-lg: 15px;
--font-xl: 18px;
--font-2xl: 24px;
```

问题不在于存在 11px 和 12px，而在于大量模块将它们用于常规说明、字段名、状态、日期和卡片内容。

本地页面抽查结果：

- 空间预约审批中，部门、时间、状态、场地等大量信息使用 12px；
- 表头和操作常用 13px；
- 描述文字使用 13px；
- 课程和物资模块 CSS 中存在大量 11–13px 自定义规则；
- 12px 经常与 `--color-text-3: #8f959e` 同时使用。

这会产生“双重弱化”：

```text
字号小 + 对比度低 = 用户认为内容模糊、不重要、难扫描
```

### 4.4 间距问题

当前虽然存在 4px 间距 Token，但模块中仍有大量硬编码：

```text
6 / 10 / 14 / 18 / 20 / 22 / 24 / 32 / 42 / 48 ...
```

主要表现：

- 相同层级的页面左右内边距不同；
- 标题到工具栏的距离不一致；
- 卡片标题、正文和底部操作分别采用不同间距；
- 卡片内部继续嵌套带边框的小卡片，间距不断缩小；
- 同一页面同时存在 12、14、16、18、20、22、24px 等多套视觉节奏；
- 大页面有时过空，小卡片内部又过挤。

### 4.5 卡片使用过度

课程详情中存在：

```text
页面容器
→ 大卡片
→ 区块卡片
→ 目标小卡片
→ 带边框按钮/资源卡片
```

每一层都通过边框和圆角强调，导致：

- 页面分割过多；
- 视线无法判断哪个层级最重要；
- 内容被压缩；
- 边框数量多于有效信息层级。

QQ 邮箱通常只为独立对象使用卡片，结构区块更多依靠留白和细分隔线。

### 4.6 信息优先级不清晰

当前页面常出现以下同级竞争：

- 页面标题与 Tab；
- 搜索与筛选；
- 状态统计与业务对象；
- 编辑入口与下一步操作；
- AI 助手与底部发布按钮；
- 右侧概览与正文中的重复信息。

不是信息数量多，而是缺乏明确的“主要对象—次要信息—辅助状态”关系。

### 4.7 内容宽度与扫描路径

- 工作台服务卡片横向过宽，单个对象信息过少；
- 课程卡片内部被分成多个小区域，阅读路径反复折返；
- 课程详情正文过宽，同时右侧栏较窄；
- 长文本缺少适合阅读的最大行宽；
- 大屏下信息被横向摊开，小字号进一步降低可读性。

### 4.8 搜索、筛选和视图控制混合

搜索和筛选并不等于页面主操作。当前多个页面把以下内容放在同一组：

```text
搜索 + 筛选 + 重置 + 视图切换 + 新建 + 导出 + 刷新
```

应拆成：

- 对象范围；
- 视图控制；
- 筛选条件；
- 对象操作。

### 4.9 滚动与固定区域不统一

当前课程详情同时存在：

- 页面整体滚动；
- 右侧概览独立滚动；
- 底部固定操作；
- 右下 AI 固定入口。

固定元素可能遮挡正文，也增加用户对“当前滚动的是哪一层”的判断成本。

### 4.10 可达性和结构完整性问题

这是本次检查发现的最高优先级问题。

#### 物资模块无法渲染

`shared/modules/material-management/material-management.js` 当前无法通过 JavaScript 语法检查。问题发生在 `templateHtml` 字符串中插入新的多行工具栏代码后，双引号字符串被真实换行打断。

实际结果：

```text
点击“物资工作台”
→ body 的 data-active 已更新
→ material-management.js 已被加载
→ 内容区为空
```

因此当前物资管理首先是“运行完整性问题”，之后才是视觉问题。

#### 空间导航入口不完整

管理员端当前“空间管理”子项只有：

- 空间预约审批；
- 班牌和大屏管理。

`空间资产管理` 模块代码存在，但在管理员端导航配置中缺失。用户无法从正常信息架构进入。

#### 模块拥有权不明确

多个模块各自监听 `body[data-active]`，并直接写入：

```js
document.querySelector('.content').innerHTML = ...
```

这是一种脆弱的内容区所有权模型。后续应至少保证：

- 路由/导航层决定唯一活动模块；
- 非活动模块只能卸载自己的实例，不能清空其他模块内容；
- 模块先通过语法检查，再进入页面回归；
- 动态模块需要统一挂载和销毁协议。

### 4.11 当前实现与目标规范对照

| 项目 | 当前实现 | 目标规范 |
|---|---|---|
| 一级页标题 | 有的缺失，有的由 Tab 代替 | 所有一级页都有独立 PageHeader |
| 详情页顶部 | 返回、标题、面板开关分散 | 返回 + 对象标题 + 状态 + 操作同处 DetailHeader |
| 创建/编辑页 | 沿用普通详情页结构 | 使用 TaskHeader、阶段导航和保存状态 |
| 搜索与筛选 | 与创建、导出、视图切换混排 | 统一进入 ControlBar，按左右语义分组 |
| 批量操作 | 长期占据页面或每页自定义 | 选择对象后显示 SelectionBar |
| 字号 | 11–13px 大量用于业务内容 | 正文 14px，元信息 13px，12px 仅限标签 |
| 文字颜色 | 小字号叠加浅灰 | 主要、次要、元信息和禁用四级明确 |
| 页面 padding | 24、32 及大量私有值混用 | 页面 32/24，区块 24，卡片 20/16 |
| 卡片 | 多层边框卡片嵌套 | 独立对象才用卡片，结构分区用留白和细线 |
| 列表 | 卡片、表格、私有行样式并存 | 统一表头、行高、单元格和操作列 |
| 右侧栏 | 同时承载目录、状态、资源和操作 | 一个右栏只承担一种核心上下文职责 |
| 模块挂载 | 多脚本竞争 `.content`，缺少统一协议 | 导航决定唯一模块，统一挂载/卸载 |
| 完整性验收 | 关注页面是否“看起来有内容” | 导航可达、字段可见、流程可操作、脚本可运行 |

## 5. 当前高完成度页面逐页问题

### 5.1 工作台

优点：

- 左侧导航结构已相对清晰；
- 服务分类完整；
- 图标系统较统一；
- 常用服务和全部服务的业务概念合理。

问题：

1. 标题与“全部服务”关系不清；
2. 搜索没有纳入稳定工具栏；
3. 首屏同时展示常用服务、全部分类、工作台服务，层级重复；
4. 服务卡片过大，信息只有标题，空间利用率低；
5. 分类使用斜杠连续分隔，扫描效率不如 Tab/Chip；
6. 页面内容直接贴近顶边，缺少稳定的 64px 页面头；
7. 同一服务在常用服务和全部服务中重复，未形成“快捷入口”和“完整目录”的视觉差异。

建议原型：**业务入口页/工作台首页**。

### 5.2 课程开发列表

优点：

- 课程对象信息较完整；
- 状态、负责人、团队、更新日期、完整度均保留；
- 卡片对象边界清楚；
- 支持卡片/列表视图。

问题：

1. 缺少显式页面标题；
2. Tab、搜索、视图切换、筛选、重置和创建动作分组不清；
3. 两个创建按钮竞争主操作；
4. 搜索与筛选不在同一控制逻辑内；
5. 卡片中同一课程名称、状态、元数据、完整度、专项归属和操作分成过多水平带；
6. 元信息字号偏小；
7. 3 列卡片在宽屏可用，但中等宽度下容易挤压；
8. “打开”和“更多”与卡片点击关系不够明确。

建议原型：**集合列表页**。

### 5.3 上传/生成教案

优点：

- 三种创建方式清楚；
- 视觉焦点单一；
- 页面内容不复杂。

问题：

1. 返回按钮、标题和说明未组成统一任务头；
2. 缺少创建流程上下文，例如“选择方式 → 导入/生成 → 核对 → 编辑”；
3. 三张卡片面积过大；
4. 卡片插图很小，正文和按钮之间留白失衡；
5. 灰色按钮既像次操作，也像禁用状态；
6. 未明确三种方式的适用场景、耗时和产出差异。

建议原型：**创建任务起始页**。

### 5.4 课程详情/课程编辑工作台

优点：

- 课程信息完整；
- 课程建设内容覆盖全面；
- 右侧概览有助于定位未完成项；
- 课程资源、评价和实施要求都已纳入；
- 具有连续编辑和发布流程。

问题：

1. 页面角色不明确：详情、编辑和发布工作台混为一体；
2. 顶部栏只有返回和标题，缺少保存状态、主要操作及稳定的右侧控制位；
3. 概览收起按钮脱离工具栏；
4. 内容块过度卡片化；
5. “编辑信息、编辑概述、编辑依据、编辑目标”样式和位置重复但未形成标准 SectionHeader；
6. 正文大量 12–13px；
7. 右侧栏同时承担导航、状态、考核和资源，职责过多；
8. 底部发布操作与 AI 助手可能遮挡内容；
9. 同一信息在主内容与右栏重复；
10. 课程内容行中按钮过多，默认态噪声大；
11. 页面滚动较长，但缺乏稳定的阶段导航和保存反馈。

建议拆为两个原型：

- **课程详情页**：只读为主，局部编辑；
- **课程编辑工作台**：阶段导航 + 编辑画布 + 上下文检查 + 保存/发布。

## 6. 统一后的产品与页面层级

### 6.1 全局层级

```text
L0 应用外壳
├─ 品牌与角色
├─ 一级业务导航
├─ 当前模块二级导航
└─ 全局账户与设置

L1 模块首页/工作台
├─ 今日任务
├─ 关键状态
└─ 模块入口

L2 对象集合页
├─ 对象范围
├─ 搜索筛选
├─ 批量操作
└─ 表格/列表/卡片

L3 对象详情页
├─ 返回与对象身份
├─ 对象状态和操作
├─ 详情内容
└─ 上下文辅助区

L4 创建/编辑任务
├─ 退出/保存
├─ 任务阶段
├─ 编辑画布
├─ 校验与辅助
└─ 提交/发布

Overlay 局部任务
├─ Drawer：查看或编辑当前对象的局部信息
└─ Modal：需要明确确认的短任务
```

### 6.2 一级页面定义

一级页面是左侧导航直接进入的模块首页或集合页。

结构：

```text
┌ PageHeader ─────────────────────────────────────────┐
│ 页面标题 + 数量/说明                  主要创建操作   │
├ ViewNav / Tabs ─────────────────────────────────────┤
│ 同级范围或业务 Tab                                  │
├ ControlBar ─────────────────────────────────────────┤
│ 搜索 + 筛选 + 重置                    排序/视图切换  │
├ ContextBar（可选）──────────────────────────────────┤
│ 当前筛选、风险提醒、批量选择状态                    │
├ MainContent ────────────────────────────────────────┤
│ 表格 / 列表 / 卡片 / 工作台                         │
└─────────────────────────────────────────────────────┘
```

规则：

- 必须有明确页面标题；
- 主操作最多 1 个，必要时使用拆分按钮或下拉；
- 搜索、筛选和视图控制归入 ControlBar；
- 统计只在能帮助决策时出现；
- 默认不使用面包屑重复左侧导航；
- 列表页优先表格/行列表，卡片只用于需要图像或对象概览的场景。

### 6.3 二级详情页面定义

二级页面对应某一个具体对象。

结构：

```text
┌ DetailHeader ───────────────────────────────────────┐
│ 返回 | 对象名称 | 状态               编辑/更多操作 │
├ ObjectMeta（可选）─────────────────────────────────┤
│ 负责人、时间、类型、编号等关键身份信息              │
├ Main + Aside ───────────────────────────────────────┤
│ 详情正文/分区内容                  目录/校验/上下文  │
└─────────────────────────────────────────────────────┘
```

规则：

- 返回、对象标题和对象操作必须在同一稳定工具栏；
- 折叠右栏的按钮属于 DetailHeader 右侧，不能单独掉落；
- 详情页默认只读，编辑应有明确进入/退出状态；
- 右栏只保留一种核心职责；
- 不在详情页重复列表页筛选；
- 不用面包屑和返回按钮同时表达同一层级。

### 6.4 创建/编辑页面定义

结构：

```text
┌ TaskHeader ─────────────────────────────────────────┐
│ 返回/关闭 | 任务名 | 保存状态        预览/提交/发布 │
├ StageNav ───────────────────────────────────────────┤
│ 基本信息 → 内容 → 资源 → 评价 → 核对                │
├ Editor ───────────────────────────────┬─────────────┤
│ 当前阶段编辑内容                      │ 校验/帮助    │
├ TaskFooter（必要时）────────────────────────────────┤
│ 上一步                         下一步/保存并继续     │
└─────────────────────────────────────────────────────┘
```

规则：

- 创建页不是普通详情页；
- 保存状态必须可见；
- 页面只能有一组主要提交操作；
- AI 助手属于辅助区或可收起浮层，不得遮挡主操作；
- 阶段导航表达流程，不与页面对象 Tab 混用。

## 7. 顶部工具栏统一规范

### 7.1 组件拆分

不再使用一个万能 Toolbar 承担全部职责。统一为：

```text
AppPageHeader
AppDetailHeader
AppTaskHeader
AppViewTabs
AppControlBar
AppSelectionBar
AppContextBar
```

### 7.2 一级页 PageHeader

```text
左：页面标题、数量、简短说明
右：唯一主操作、最多一个次操作、更多
```

推荐高度：

```text
单行：64px
带说明：72–80px
```

### 7.3 一级页 ControlBar

```text
左：搜索、常用筛选、更多筛选、重置
右：排序、密度、列表/卡片视图
```

规则：

- 搜索默认宽 280–320px；
- Select 高度统一 36px；
- 常用筛选直接显示，低频筛选进入“更多筛选”；
- 重置仅在存在筛选条件时高亮可用；
- 创建按钮不进入筛选组。

### 7.4 选择状态 SelectionBar

选择对象后，用选择栏替代或覆盖普通 ControlBar：

```text
已选 6 项 | 批量操作                        取消选择
```

不要让批量按钮在未选择对象时长期占据工具栏。

### 7.5 详情页 DetailHeader

```text
左：返回、对象名称、状态
右：主要对象操作、更多、右栏开关
```

禁止：

- 把右栏开关放到第二行左侧；
- 只放“返回”而没有对象身份；
- 把列表筛选带入详情页；
- 在正文卡片内重复整个对象的主操作。

### 7.6 编辑页 TaskHeader

```text
左：保存并返回/退出、任务标题
中或标题旁：保存中/已保存/有未保存修改
右：预览、提交/发布
```

## 8. 项目文字规范

鉴于客户已多次反馈文字太小，本项目正文基准应高于当前实现。

### 8.1 推荐字号

| Token | 字号/行高 | 用途 |
|---|---|---|
| `display` | 28/38 | 数据大屏或极少量大标题 |
| `page-title` | 20/28 | 一级页面标题 |
| `detail-title` | 22/30 | 对象详情标题 |
| `section-title` | 16/24 | 区块标题、卡片标题 |
| `body` | 14/22 | 默认正文、表格、列表 |
| `body-strong` | 14/20 | 关键对象名称、按钮 |
| `meta` | 13/20 | 日期、部门、编号等元信息 |
| `caption` | 12/18 | 标签、角标、极短辅助信息 |

### 8.2 强制规则

1. 正文、表格内容、表单标签不得默认使用 12px；
2. 12px 只允许用于标签、角标和非关键元信息；
3. 11px 只允许用于特殊小标识，不得用于业务句子；
4. 普通按钮文字统一 14px；
5. 输入框文字统一 14px；
6. 页面说明最小 14px；
7. 13px 文字不得同时使用过浅颜色；
8. 中文正文行高不得低于 1.5。

### 8.3 推荐文字颜色

```css
--text-primary:   #1D2129; /* 标题、关键值 */
--text-secondary: #4E5969; /* 正文、描述 */
--text-tertiary:  #6B7785; /* 元信息 */
--text-placeholder:#86909C; /* 占位、禁用提示 */
--text-disabled:  #C9CDD4;
```

现有 `#8f959e` 不宜继续用于 12px 的重要时间、部门、状态和字段值。

## 9. 间距与布局规范

### 9.1 基准网格

只使用：

```text
4 / 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48
```

除特殊几何对齐外，禁止新建 6、10、14、18、22、26、28px 等页面间距。

### 9.2 页面间距

| 场景 | 建议值 |
|---|---:|
| 页面左右内边距 | 32px；窄屏降至 24px |
| PageHeader 水平内边距 | 32px |
| 主内容上下内边距 | 24px |
| 一级区块间距 | 24px |
| 同一区块内部模块 | 16px |
| 紧密控件间距 | 8px |
| 图标与文字 | 6–8px |

### 9.3 卡片间距

| 卡片类型 | 内边距 |
|---|---:|
| 标准内容卡片 | 20px |
| 紧凑列表卡片 | 16px |
| 大型详情区块 | 24px |
| 卡片 Header | 16px 20px |
| 卡片 Footer | 12px 20px |

规则：

- 同一张卡片内只使用一套水平内边距；
- Header、Body、Footer 的左右边线必须对齐；
- 禁止默认嵌套三层带边框卡片；
- 信息分区优先使用标题、留白和分隔线。

### 9.4 内容宽度

- 列表和表格可以铺满；
- 阅读型正文最大宽度建议 960–1120px；
- 表单主体最大宽度建议 960px；
- 右侧辅助栏建议 280–360px；
- 主从分栏主区域不得小于可用宽度的 65%。

## 10. 表格、列表与卡片规范

### 10.1 表格

建议：

```text
表头高度：44px
普通行：52–56px
多行信息行：64–72px
单元格水平 padding：16px
表头和正文：14px
辅助信息：13px
```

列顺序：

```text
选择 → 对象身份 → 核心业务信息 → 状态/时间 → 行操作
```

### 10.2 行操作

- 高频且明确的单一操作可直接显示；
- 两个以上低频操作放入“更多”；
- 删除不应与普通查看同等强调；
- 整行可点击时，仍需保留明确的可访问入口；
- Hover 才出现的操作不得是唯一入口。

### 10.3 卡片

只有以下情况使用卡片：

- 对象需要图片识别；
- 对象包含 2–3 组不可拆分的概览信息；
- 对象之间需要明显独立；
- 卡片本身可以作为完整点击目标。

如果内容天然是同构字段，优先使用表格或行列表。

## 11. 三类页面的整改策略

## 11.1 A 类：完成度高，统一规范

范围示例：

- 工作台；
- 课程开发；
- 课程详情；
- 新建课程；
- 课程编辑工作台。

处理原则：

1. 不推翻业务内容；
2. 先重新归类页面原型；
3. 替换顶部结构；
4. 提升文字可读性；
5. 统一间距与卡片层级；
6. 抽取可复用组件；
7. 清理模块内重复 CSS。

验收重点：

- 一级页、详情页和编辑页能一眼区分；
- 相同语义的工具始终位于相同位置；
- 普通正文不再小于 14px；
- 主操作唯一；
- 卡片嵌套不超过两层。

## 11.2 B 类：完成度低，但数据没有丢失

范围示例：

- 空间预约审批；
- 空间资产管理；
- 空间相关列表与管理页。

处理原则：

1. 保留现有数据结构与字段；
2. 补齐导航入口；
3. 使用 A 类页面形成的 Header、ControlBar、Table/List 和 Drawer；
4. 统一列表行高、字号、筛选栏和操作列；
5. 不为每个空间页面重新设计一套样式。

空间模块建议架构：

```text
空间管理
├─ 空间预约审批
├─ 空间资产管理
└─ 班牌和信息大屏
```

空间预约审批应使用：

```text
PageHeader
→ 状态 Tabs
→ ControlBar
→ 风险 ContextBar
→ 申请列表/表格
→ 审批 Drawer
```

## 11.3 C 类：完成度低，并存在信息/功能损失

范围示例：

- 物资管理。

处理原则：

1. 旧版文件是业务信息基线；
2. 当前项目视觉规范是界面基线；
3. 不能用“页面更简洁”为理由删除字段、状态、记录或流程；
4. 重构前先建立业务对象和字段清单；
5. 完成字段对照和交互对照后才能修改页面；
6. 先恢复正常渲染，再进行视觉重构。

### 12.5 当前版本的真实状态

当前源码中仍可找到大量旧版业务字段和 Mock Data，因此不能简单判断为“源码数据全部删除”。

但从用户可用界面看，当前存在三类实际损失：

1. **渲染损失**：脚本语法错误导致内容区空白；
2. **入口损失**：部分业务入口在不同角色导航中不一致；
3. **表达损失**：即使数据仍在源码中，如果默认列表、详情或流程无法展示，用户仍然等同于失去信息。

因此物资管理的重构验收必须以“页面可达、信息可见、流程可操作”为准，不能以“字段还在 JS 文件中”为准。

## 15. 验收标准

### 15.1 结构

- [ ] 每个页面能明确归属一级页、详情页或编辑任务页；
- [ ] 顶部导航、对象操作、筛选和视图控制不混组；
- [ ] 主操作最多一个；
- [ ] 详情页返回、对象标题和操作在同一 Header；
- [ ] 右栏只承担一种核心职责；
- [ ] 页面只有一个视觉主角。

### 15.2 文字

- [ ] 正文与表格主体不小于 14px；
- [ ] 元信息不小于 13px；
- [ ] 12px 只用于标签和极短辅助信息；
- [ ] 普通按钮和输入框文字为 14px；
- [ ] 重要信息不使用低对比度灰色；
- [ ] 中文正文行高不低于 1.5。

### 15.3 间距

- [ ] 页面左右内边距统一；
- [ ] 区块间距只使用规范 Token；
- [ ] 卡片 Header/Body/Footer 左右对齐；
- [ ] 无三层带边框卡片嵌套；
- [ ] 表格行高和单元格 padding 统一。

### 15.4 完整性

- [ ] 所有导航入口可达；
- [ ] 模块脚本通过语法检查；
- [ ] 页面切换无空白；
- [ ] 物资旧版字段均有去向；
- [ ] 物资旧版流程均有入口；
- [ ] 数据存在于源码不算完成，必须在 UI 中可见或可进入；
- [ ] 不因视觉简化删除业务状态、记录和关键操作。

### 15.5 回归

- [ ] 1280×720 可操作；
- [ ] 1440×900 信息完整；
- [ ] 1920×1080 不因过宽导致文字分散；
- [ ] 页面缩放 100% 时无需依赖浏览器放大；
- [ ] 固定操作和 AI 浮层不遮挡内容；
- [ ] 每个模块在切换角色和导航后都能正确挂载与卸载。

## 16. 最终设计原则

后续所有页面统一遵守以下原则：

1. 统一的是页面语法，不是把所有页面做成同一张模板；
2. 先保全业务信息，再做视觉精简；
3. 页面标题、对象操作、视图控制必须分工明确；
4. 主要信息使用足够大的字号和足够深的颜色；
5. 留白表达层级，边框只表达真正独立的对象；
6. 一级页服务于浏览和管理，详情页服务于理解对象，编辑页服务于完成任务；
7. 工作台只聚合需要立即判断和行动的信息；
8. 任何“简洁”都不能以删除业务字段、状态、记录和流程为代价；
9. 设计规范必须落到共享组件和验收检查，不能只停留在 Markdown；
10. 先形成一个可信样板，再复制到全系统。
